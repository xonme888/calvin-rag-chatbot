# 002. 비용 의식 + 점진적 빌드 (포트폴리오 시연)

## 상황
LLM 호출이 인덱싱·평가·시연 곳곳에 들어가는 RAG 시스템에서 "전체를 무지성으로 인덱싱"하는 게 기술적으로 가능하지만 시니어 엔지니어 시각에선 영리해 보이지 않는다.

구체 사례: 칼빈 강요 1,251p → 5,657청크 LLMGraphTransformer 추출 = ₩2,000 + 3시간. 이 비용 자체는 작지만 **포트폴리오에 보여주기엔 의식 없는 빌드**.

## 결정
**3단계 점진적 빌드**:
1. 사전 비용 추정 (LLM 호출 없이 청크 수 × 단가)
2. 작은 샘플(50청크 ≈ ₩17)로 스키마 동작 검증 → 이상 시 보정
3. 정확한 분량(예: 단원별 30청크 균등 = 150청크 ≈ ₩50)으로 본 인덱싱
4. README/시연 화면에 **확장 가능성 추정표 명시** ("동일 파이프라인으로 전체 1,251p 인덱싱 가능, 추정 ~₩175")

## 근거
- "₩2,000 무지성 인덱싱" vs "₩70 + 확장 가능성 명시"는 **같은 시연 결과**를 만들 수 있다 — 단지 후자가 절제된 의사결정처럼 보임
- 시니어 면접에서는 결과보다 *결정 과정*이 평가 대상
- 단계적 인덱싱은 스키마 버그(예: LLM이 엉뚱한 노드 타입 만듦)를 일찍 잡아주는 안전장치 — 비용 회수
- 추정 코드(`estimate_cost`)를 시각화하면 비용 의식이 코드에 녹아 있음을 어필 가능

## 적용 방법
1. 인덱싱 스크립트에 `--sample N`, `--balanced N`, `--full` 3모드 항상 제공
2. 각 모드 실행 전 비용 추정 + confirm 프롬프트
3. 추정 함수(`estimate_cost(chunks)`)를 별도 함수로 노출 — 시연 시 결과 화면에 표시
4. README에 "확장 가능성" 섹션: 현재 인덱싱 분량 / 전체 분량 / 추정 비용 차이

## 사례
- 50청크 샘플 (₩17) 검증 → 단원별 30청크 균등 보강 (₩35) → 누적 ₩52
- 검증 결과로 단원별 커버리지 부족 발견 → 보강 결정에 활용
- README 면접 어필 문구: "비용 ₩52로 5단원 시연. 동일 파이프라인으로 전체 1,251p 인덱싱 시 ~₩175."
