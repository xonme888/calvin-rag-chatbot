# 001. 의사결정은 트레이드오프 표로

## 상황
선택지가 둘 이상이고 비용/시간/효과의 차원이 서로 충돌할 때.
구체적으로:
- KG 인덱싱 비용 옵션 5가지 (₩30~₩290)
- repo 분리 (2개 vs 3개 vs 1개)
- KG 어댑터 추상화 정도 (단순 토글 vs Port+Adapter vs 다중 백엔드)
- Neo4j 호스팅 (로컬 Docker vs Aura Cloud)

## 결정
"어느 게 가장 좋은가?"를 묻기 전에 **표로 옵션을 늘어놓고 차원별로 비교**한다. 표가 없으면 권장하지 않는다.

```
| 옵션 | 비용 | 시간 | 시연 효과 | 어필 포인트 |
|---|---|---|---|---|
| A | ₩30 | 1일 | 약함 | "절대 저비용" |
| B | ₩170 | 2일 | 중 | "균형" ★ |
| C | ₩290 | 3일 | 강함 | "풍부함" |
```

## 근거
- 사람은 단일 정답을 추구하기 쉬우나, 엔지니어링은 거의 항상 **포기한 차원**이 있다. 표는 그 포기를 명시하게 한다
- ★ 표시로 추천 옵션을 마킹하되 사용자가 다른 선택을 할 여지를 남긴다 (CLAUDE.md 시스템 가이드 "recommendation + tradeoff" 형식)
- 비용을 항상 **절대값(₩, 시간, 디스크)** 으로 환산해 두면 의사결정이 빠르다 — 추상적 "약간 비싸다"는 결정 못 내리게 함

## 적용 방법
1. 옵션을 3~5개로 정리 (너무 많으면 의사결정 마비)
2. 차원 4개 이내: 비용·시간·효과·어필포인트 정도가 적정
3. ★ 추천 표시 + 추천 이유 1~2줄
4. 다른 의견 받을 여지 명시: "다른 안 선호?"

## 사례
- KG 비용 5옵션 (옵션 A~E) → 옵션 C 채택, 사용자가 추가 의견 후 옵션 1 균형안으로 보강
- 어댑터 4옵션 (단순/포트2/포트1+Config/다중백엔드) → 포트1+Config 채택
- 단원 선정 3안 (타이트 ₩93 / 균형 ₩175 / 풀 ₩290) → 균형안 채택
