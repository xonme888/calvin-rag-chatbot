# PRD: 데이터/UX 안전망 (재시도 버튼 · 저장소 한도 · 라우터 진화 · 자동 제목)

## 1. 배경 / 문제

라우팅이 자동(`mode='auto'`) 인 데도 사용자가 "이 답이 다른 모드였다면 어땠을까" 를 시험할 길이 없다. `rag_core/router.py:42-70` 의 `route_question` 은 키워드 휴리스틱이고 `metadata.routed_mode` 까지 노출되지만, 답변 카드 옆에서 **즉시 다른 모드로 같은 질문을 재실행** 하는 동선은 비어 있다. 잘못된 라우팅을 사용자가 한 번에 교정할 수 없다.

저장소 한도도 잠재 문제다. `web/lib/sessionStore.ts:128-131` 에서 전체 sessions 를 매번 `JSON.stringify` 후 `localStorage.setItem` 한다. KG subgraph (`metadata.subgraph`) 와 도구 결과(PRD-1) 가 누적되면 단일 도메인 (5MB) 를 초과할 수 있다. PRD-2 가 진행되면 무관해지지만, PRD-2 가 미뤄지면 단일 디바이스 안에서도 깨진다.

라우터 진화도 미뤄둔 항목이다 (task #52, "[추후] 라우터 진화 — audit log + 재시도 버튼 + LLM swap"). audit 인프라(`infra/observability.py:trace_event`, `router.decide` 이벤트) 가 이미 깔려 있어 다음 단계로 갈 준비는 됐다.

마지막으로 세션 자동 제목 — `sessionStore.ts:71-77` 의 `deriveTitle` 은 첫 사용자 질문 30자 truncate 다. 사이드바에서 "신학…" 같이 잘리는 케이스가 흔하다.

본 PRD 는 **PRD-1, PRD-2 가 들어와도 깨지지 않는 작은 안전망 4개** 를 묶는다.

## 2. 목표

- 사용자가 라우팅 결과를 한 번에 다른 모드로 재시험할 수 있다.
- 저장소 한도를 넘지 않는다 (PRD-2 와 무관하게).
- 라우터의 다음 진화 방향을 본 PRD 에서 결정한다 (audit 활용).
- 사이드바 세션 제목이 한눈에 식별 가능하다.

## 3. 비-목표

- 인증/다기기 동기화 (PRD-2).
- 새 도구/이미지 입력 (PRD-1).
- RAGAS 평가 자동화 / `experiments/eval/` 변경.
- 모바일 전용 레이아웃.

## 4. 사용자 시나리오 / BDD

- Given 사용자가 "어거스틴과의 관계" 질문에 KG 모드 답변을 받은 후
  When 답변 헤더 옆 "다른 모드로 재시도" 메뉴에서 "Hybrid" 를 고르면
  Then 같은 질문이 Hybrid 로 즉시 재실행되고, 두 답변이 같은 세션 안에 시간순으로 남는다.

- Given KG subgraph 가 누적되어 sessions 페이로드가 4MB 를 넘는 상황에서
  When 사용자가 새 질문을 보내면
  Then 저장 실패 토스트가 뜨지 않고, IndexedDB 어댑터로 영속화된다 (또는 PRD-2 채택 시 서버로).

- Given 라우터가 `kg` 로 결정한 질문에서
  When trace audit 로그를 확인하면
  Then `decided=kg`, `matched_hint=...` 와 함께 사용자가 재시도 버튼으로 변경한 모드(있다면)가 같은 trace_id 로 묶여 기록된다.

- Given 사용자가 새 세션에 "칼빈은 예정을 어떻게 정의했는가" 라고 첫 질문을 입력하면
  When 답변이 끝나면
  Then 사이드바 제목이 "예정 정의" 같은 요약(또는 30자 truncate) 으로 자동 갱신된다.

## 5. 결정해야 할 사항

### 결정 1 — 라우터 진화 방향

| 옵션 | 비용 | 가치 | 위험 | 추천 |
|---|---|---|---|---|
| "다른 모드로 재시도" 버튼만 | 반나절 | 사용자가 직접 교정 가능, audit 로 미스 라우팅 패턴 수집 | 자동화는 미뤄둠 | ★ |
| cheap LLM 즉시 swap (`router.py` 의 함수만 교체) | 1~2일 + 호출 비용 | 정확도 향상 기대 | 라우팅 latency +α, cheap LLM 도 가끔 틀림 | |
| 휴리스틱 키워드만 보강 (audit 분석 후 cron 으로 키워드 추가) | 1일 | 비용 0 | 룰 폭주, 유지보수 부담 | |

비고: ★(재시도 버튼) 채택 시 audit 데이터가 모이고, 그것을 근거로 다음 단계에서 cheap LLM swap 또는 키워드 보강을 결정한다. 두 단계 분리가 합리적.

### 결정 2 — 저장소 한도 대응

| 옵션 | 비용 | 가치 | 위험 | 추천 |
|---|---|---|---|---|
| W1: localStorage → IndexedDB 어댑터 swap (`useSessions` 인터페이스 유지) | 1일 | 5MB 한도 회피, PRD-2 와 무관하게 즉시 효과 | PRD-2 가 들어오면 작업 일부 무용 | ★ (PRD-2 가 한 달 이상 미뤄질 때) |
| 가지치기 (오래된 세션 자동 보관/삭제) | 0.5일 | 단순 | 사용자 데이터 의도치 않은 손실 | |
| 아무것도 안 함, PRD-2 만 기다림 | 0 | 작업 0 | PRD-2 가 늦으면 사용자 막힘 | |

조건부 추천: PRD-2(다기기 동기화) 가 본 PRD 직후 진행되면 W1 스킵. 그렇지 않으면 W1 채택.

### 결정 3 — 세션 자동 제목 품질

| 옵션 | 비용 | 가치 | 위험 | 추천 |
|---|---|---|---|---|
| 30자 truncate 그대로 | 0 | 0 비용 | 사이드바 가독성 낮음 | |
| cheap LLM 자동 요약 (첫 질문 + 첫 답변 보고 8~12자) | 답변 1건당 cheap LLM 1회 추가 | 사이드바 가독성 큼 | 비용 작음, 품질 편차 | ★ |

## 6. 기능 요건

- 답변 헤더 옆 "다른 모드로 재시도" 드롭다운 — 현재 모드를 제외한 옵션 노출. 클릭 시 같은 세션에 같은 질문을 다른 모드로 재요청.
- 재시도 결과는 같은 세션의 다음 메시지로 추가되며, 사용자에게 "재시도 결과 (모드: …)" 배지 노출.
- 라우터 audit 에 `retry_from_mode`, `retry_to_mode` 필드 추가 (사용자가 재시도한 경우만).
- (조건부) `useSessions` 의 영속화 어댑터를 IndexedDB 로 swap. 인터페이스 변경 없음.
- (선택) 첫 답변 종료 시점에 cheap LLM 으로 세션 제목 1회 갱신. 사용자가 제목을 직접 편집한 후에는 자동 갱신 비활성.

## 7. 성공 지표 (정량)

- "다른 모드로 재시도" 사용 후 사용자 만족도 (수동 5건 시범) ≥ 80% (= 답변이 더 적합 또는 비교 가치 있음).
- 단일 디바이스에서 sessions 페이로드 10MB 누적 시 저장 실패 0건 (W1 적용 시).
- 라우터 audit 의 `decided` 분포가 30일 누적으로 사이드바에서 한 줄 통계로 확인 가능 (운영자 가시성).
- 사이드바 세션 제목 평균 길이 8~14자, "새 대화" 잔여율 ≤ 5%.

## 8. 의존 / 영향 / 회귀 위험

- **의존**: 본 PRD 는 PRD-1/PRD-2 와 독립 진행 가능. 단 W1 의 채택 여부는 PRD-2 일정에 따라 달라진다.
- **영향**: 재시도 audit 데이터가 쌓이면 결정 1 의 다음 단계(cheap LLM swap / 키워드 보강) 결정 근거가 된다. PRD-2 의 trace `user_id` 와 결합하면 사용자별 라우팅 정확도까지 측정 가능.
- **회귀 위험 (저)**: 재시도가 같은 세션에 메시지 2개를 더하므로, PRD-2 의 메시지 단위 영속화 스키마와 시간순 정렬이 일치해야 한다.
- **회귀 위험 (저)**: IndexedDB 어댑터 swap 시 SSR/CSR mount 타이밍이 현재 `sessionStore.ts:106-125` 와 동일한 패턴을 유지해야 한다 (mismatch 회피).
- **회귀 위험 (저)**: 자동 제목 LLM 호출이 SSE 답변 종료 콜백에 묶이면 실패 시 답변 노출이 지연되지 않도록 비동기 분리 필요. 세부 TRD.

비고: 자동 제목 cheap LLM 모델 선택, IndexedDB 라이브러리 선택(`idb` vs 자체 wrapper) 은 TRD 에서 확정.
